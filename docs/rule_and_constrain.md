# Pong 1v1 Online - Rules and Constraints

## 1. ข้อมูลทั่วไป (Overview)
- **ประเภทเกม**: Online 1v1 Pong แบบ Real-time
- **สัดส่วนหน้าจอ**: แนวตั้ง (Vertical) อัตราส่วน 2:3 โดย Container ปรับขนาดให้พอดีกับหน้าจอ (Responsive Fit) ในทุกอุปกรณ์
- **ธีมกราฟิก**: สีดำ-ขาว (Black Background, White Elements) ในสไตล์ Pixel 8-bit (ใช้ Pixel font และลูกบอลทรงสี่เหลี่ยม)
- **แพลตฟอร์ม**: รองรับการเล่นผ่าน Web Browser ทั้งบน PC และอุปกรณ์ Mobile
- **Frame Rate & Interpolation**: Backend ประมวลผลและส่งข้อมูลสถานะที่ความถี่ 30 FPS จากนั้น Frontend ดำเนิน Client-Side Prediction (Interpolation) เพื่อให้การแสดงผลลื่นไหลในระดับเสมือน 60 FPS

## 2. ระบบห้องและการจับคู่ (Room & Matchmaking)
- **สถานะผู้เล่น**:
  - `Creator`: ผู้สร้างห้อง ได้รับรหัสห้อง (UUID) สำหรับแชร์ให้ผู้เล่นฝ่ายตรงข้าม
  - `Joiner`: ผู้เข้าร่วม นำ UUID มาป้อนและยืนยันการเข้าร่วมเพื่อเริ่มเกมแบบ 1-1
- **ขั้นตอนการทำงานของ Lobby**:
  1. หน้าหลักแสดงปุ่ม 2 ปุ่ม: `Create` และ `Join`
  2. เมื่อกด `Create`: ระบบสร้าง UUID แบบสุ่ม → แสดงปุ่ม Copy/Share → เปลี่ยนสถานะเป็น Waiting (รอผู้เล่นฝ่ายตรงข้าม)
  3. เมื่อกด `Join`: แสดงช่องกรอก UUID → ผู้เล่นวาง UUID → กด `Join` เพื่อยืนยัน
  4. เมื่อผู้เล่นทั้งสองฝ่ายเชื่อมต่อกันในห้องสำเร็จ ระบบจะนำทางเข้าสู่สนามประลอง (Arena)

## 3. กติกาการเล่น (Gameplay Rules)
- **มุมมองผู้เล่น**: ผู้เล่นในเครื่องตนเอง (Local Player) จะแสดงอยู่ **ด้านล่างเสมอ** ในขณะที่คู่แข่ง (Opponent) จะแสดงอยู่ด้านบน
- **ขอบเขตสนาม**: ขอบซ้ายและขอบขวาของสนามสะท้อนลูกบอลได้ ขอบด้านหลัง Pad ของผู้เล่นแต่ละฝั่งถือเป็นเขตเสียแต้ม หากลูกบอลผ่านออกไปจะนับเป็นคะแนนของฝ่ายตรงข้ามทันที (ตามกติกาดั้งเดิมของ Atari)
- **คะแนน (Score)**: นับสะสมแบบไม่จำกัดรอบ (Endless Game) พร้อมแสดงคะแนนของแต่ละฝ่ายอย่างชัดเจนในสไตล์ Atari
- **เงื่อนไขการเสิร์ฟ**:
  - เริ่มเกมครั้งแรก: `Creator` ได้รับสิทธิ์เสิร์ฟก่อนเสมอ
  - ระหว่างการแข่งขัน: ฝ่ายที่เสียแต้มจะได้รับสิทธิ์เสิร์ฟในรอบถัดไปสลับกันไป
  - ลักษณะการเสิร์ฟ: ลูกบอลจะคงอยู่ที่จุดกึ่งกลาง Pad ของผู้มีสิทธิ์เสิร์ฟ จนกว่าผู้เล่นจะกด "Action (เสิร์ฟ)" เพื่อดีดลูกบอลออกไป

## 4. ระบบฟิสิกส์และการเคลื่อนที่ (Physics & Movement)
- **มุมการสะท้อน (Bounce Angle - Arkanoid-style)**:
  - มุมสะท้อนของลูกบอลเมื่อกระทบ Pad ถูกกำหนดด้วย **ตำแหน่งจุดกระทบบน Pad** เสมอ (ไม่ใช่การสะท้อนตามมุมตกกระทบแบบปกติ)
  - *ขอบซ้ายสุดของ Pad*: สะท้อนในมุม -75 องศา (อิงจากแกน Y แนวตั้ง)
  - *จุดกึ่งกลางของ Pad*: สะท้อนในมุม 0 องศา (พุ่งตรงแนวตั้ง)
  - *ขอบขวาสุดของ Pad*: สะท้อนในมุม +75 องศา
- **ระบบ Power Hit**:
  - หากผู้เล่นกดปุ่ม Power Hit **ในจังหวะที่ถูกต้อง** (ขณะลูกบอลอยู่บน Pad หรืออยู่ในช่วงก่อนกระทบ Pad เล็กน้อย) ลูกบอลจะสะท้อนกลับด้วยความเร็วสองเท่า (**Speed ×2**)
  - *มุมสะท้อน*: เมื่อ Power Hit เปิดใช้งาน มุมการสะท้อนจะ **แคบลงครึ่งหนึ่ง** (เช่น มุม 75 องศาปกติจะเหลือ 37.5 องศา)
  - ระบบต้องแสดง **Effect ทางภาพและแอนิเมชัน** เพื่อยืนยันว่า Power Hit ประสบผลสำเร็จ
  - Power Hit มีผลเฉพาะ *การกระทบครั้งแรก* เท่านั้น ความเร็วพิเศษจะคงอยู่จนกว่าลูกบอลจะกระทบ Pad ของฝ่ายตรงข้าม หากฝ่ายตรงข้ามรับลูกได้โดยไม่ได้กด Power Hit สวนกลับ ความเร็วบอลจะกลับสู่ **Speed ×1** ทันที

## 5. การควบคุม (Controls)
- **PC (Mouse)**:
  - เลื่อนเมาส์ซ้าย-ขวาภายในสนาม: ควบคุมตำแหน่ง Pad ให้ตามตำแหน่งเมาส์
  - คลิกเมาส์: เสิร์ฟลูก (ขณะ Serving) / สั่งการ Power Hit (ขณะ Playing)
- **อุปกรณ์สัมผัส (Touch)**:
  - ลากนิ้ว (Drag/Swipe) ซ้าย-ขวาภายในสนาม: ควบคุมตำแหน่ง Pad ให้ตามตำแหน่งนิ้ว
  - แตะ (Tap): เสิร์ฟลูก (ขณะ Serving) / สั่งการ Power Hit (ขณะ Playing)

## 6. ข้อกำหนดทางเทคนิคและสถาปัตยกรรม (System & Architecture)
- **เทคโนโลยีที่ใช้ (Stack)**:
  - **Runtime & PM**: `Bun`
  - **Frontend (`pong-fe`)**: React + Vite + Tailwind CSS
  - **Backend (`pong-be`)**: Express + TypeScript
- **เครือข่าย (Network)**:
  - ใช้ **Native WebSockets เท่านั้น** (ไม่อนุญาตให้ใช้ Socket.io) ทั้งฝั่ง Client และ Server
- **โครงสร้างไฟล์ (Structure)**:
  - จัดโครงสร้างโปรเจกต์แบบ **Feature-based Structure** ในทั้งสองระบบ เพื่อแบ่ง Domain อย่างชัดเจน
- **Shared Constants (ข้อมูลร่วมกัน)**:
  - ต้องมีกลไกประกาศค่าคงที่ (Event Names, Action Types, Config Numbers) แยกเป็นไฟล์อิสระ เพื่อให้ Frontend และ Backend ใช้ข้อมูลชุดเดียวกันได้อย่างสมบูรณ์
- **ความทนทานต่อความผิดปกติทางเครือข่าย (Network Resilience)**:
  - **Grace Period**: กำหนดระยะเวลา 1 นาที สำหรับกรณีที่ผู้เล่นฝ่ายใดฝ่ายหนึ่งสูญเสียการเชื่อมต่อโดยไม่ตั้งใจ ระบบจะรักษาสถานะเกมไว้เพื่อรอการเชื่อมต่อกลับ หากเกินกำหนดเวลาจึงจะตัดสินให้ฝ่ายที่ไม่เชื่อมต่อเป็นผู้แพ้และสิ้นสุดเกม
